# World Studio 面试讲解提纲

## 30 秒项目介绍

World Studio 是一个本地优先的世界观创作工具。它不是单纯的表单 CRUD：同一个角色或地点会同时被富文本、关系图谱、地图、时间线、画布和素材库引用。我主要解决的是复杂前端状态如何建模、本地数据如何可靠保存，以及大型交互模块如何按需加载并在失败时降级。

## 5 分钟演示顺序

1. 从概览进入最近编辑条目，展示全局搜索和模块导航。
2. 编辑正文，插入标题、条目引用或素材图片，刷新页面验证正文仍在。
3. 在关系、地图或画布中展示同一条目的跨模块引用。
4. 创建第二个世界并切换，证明数据隔离。
5. 导出备份，最后展示测试与 CI，而不是继续浏览所有按钮。

## 建议重点讲的三个问题

### 1. 如何降低正文丢失风险？

- 正文不是只在离开编辑器时保存，而是有独立草稿层。
- IndexedDB 写入经过队列，避免旧异步写入覆盖新状态。
- 条目保留修订版本，异常 HTML 会先规范化。
- Tiptap 单独设置错误边界，失败后仍可使用兼容编辑器。

可以主动说明：这仍不是服务端强一致，也不能替代跨设备同步；它解决的是单浏览器本地创作的可靠性。

### 2. 多世界为什么不是给每个 Zustand store 再加一个 worldId？

当前方案用世界注册表加版本化工作区快照。切换时先保存当前世界，再恢复目标世界的完整领域状态。这样能保留现有 feature store 的清晰边界，也让导入导出复用同一份 schema。代价是世界切换属于较重操作；如果未来需要多世界同时查询，应改成真正按 worldId 规范化的数据层。

### 3. 为什么包里仍有较大的 Tiptap 和 Cytoscape？

它们是专业编辑和图布局能力的成本，关键不是把依赖名称藏起来，而是让它们不进入无关页面的首屏路径。Tiptap 只在进入编辑状态时动态加载，图谱依赖使用独立 vendor chunk，CI 还设置了 gzip 预算。若进一步优化，应基于部署后的 Coverage 与 Core Web Vitals，而不是盲目拆分。

## 高频追问与回答方向

| 问题 | 回答重点 |
| --- | --- |
| 为什么选 IndexedDB？ | 数据包含大量结构化记录和二进制资源，容量与异步事务能力比 localStorage 合适 |
| 为什么用 Zustand？ | feature store API 简洁，适合本地领域动作；持久化通过自定义 adapter 与 UI 状态解耦 |
| 如何处理删除？ | 集中式领域级联，不让各页面各删一部分；同时清理跨模块 ID 引用 |
| 如何验证迁移安全？ | 备份包含 format/version，导入先验证枚举、ID 和引用，再替换现有工作区 |
| 如何避免一个页面报错后全站失效？ | 路由级错误边界按 location 重置；编辑器还有更细粒度的恢复边界 |
| 移动端做了什么？ | 入口重新组织、弹层避让、触控尺寸、键盘焦点、主题与语言入口，而非只缩小桌面布局 |
| 还有什么不足？ | 无云同步/协作；缺真实用户性能监控；超大数据量尚需虚拟列表；无障碍还需正式读屏审计 |

## 不要这样介绍

- 不要逐页念功能列表。
- 不要声称 IndexedDB 是“数据库后端”或已经支持强一致。
- 不要回避兼容编辑器；它恰好能体现故障隔离设计。
- 不要把所有技术选择说成最佳方案，要说明当前产品约束和未来切换条件。

## 面试前检查清单

- 控制台没有未解释的错误或重复警告。
- 准备一份可导入的演示世界备份，避免现场手工造数据。
- 本地提前运行 `npm run lint && npm test && npm run build && npm run check:bundle`。
- 浏览器缩放保持 100%，分别准备桌面和窄屏演示。
- 能指出一项尚未完成的工程工作，并给出可验证的下一步。
